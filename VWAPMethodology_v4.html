Proposed methodology · VWAP white paper

Earn the benchmark before blending it.

This proposal defines a defensible benchmark stack for H100 compute pricing. The core benchmark is Completed Rental VWAP, supported by independently measured uptime, bounded offer-quality weights, explicit carryforward limits, source-rights controls, and public methodology governance.

01 · Revised thesis

The composite is not the first product. The component benchmark stack is.

A blended price is only useful after its ingredients are trusted. v0.1 should publish the component series with quality metadata and leave the composite marked as experimental. This keeps the product honest in thin markets: users can see whether the signal comes from completed rentals, Web2 anchors, or offers.

03 · Minimum viable benchmark

Define the bucket narrowly enough to defend it.

SO-H100-DLV-v0.1 bucket definition
chip_model:          NVIDIA H100
memory:              80GB
topology:            explicit field: SXM | PCIe | unknown
region:              North America, with sub-region metadata
market_type:         on-demand / short-term rental
excluded_types:      bulk contract, reserved term, enterprise commitment
unit:                USD per verified GPU-hour
publication_status:  dashboard/API only
settlement_ready:    false by default
Why this specified: a single large discounted bulk deal can overwhelm 50 smaller rentals in a naive VWAP. Rental type stratification prevents a batch enterprise discount from masquerading as spot/on-demand market clearing price.
04 · Verification layer

“Verified GPU-hours” must be admitted by rule, not adjusted by discretion.

This methodology uses verification tiers as a binary admission gate: a rental either satisfies the minimum proof threshold and enters VWAP at its raw verified GPU-hours, or it is excluded and logged.

Level 1

Economic proof

Lease exists, payment settled, duration observable. Useful, but insufficient alone for settlement-grade pricing.

admitted only for research mode
Level 2

Telemetry proof

Provider telemetry confirms uptime, job lifecycle, GPU allocation, and basic utilization signals.

minimum v0.1 admission target
Level 3

Hardware attestation

Cryptographic or trusted hardware evidence confirms that the claimed GPU class served the rental.

institutional target
Level 4

Buyer confirmation

Customer-side completion evidence, workload output, or buyer-side verification confirms usable delivery.

highest confidence
Verified GPU-hours admission rule
usable_hours_i =
  observed_rental_duration_i × uptime_ratio_i

verification_tier_i =
  max_proof_level_satisfied_by_record_i

admission_rule:
  if verification_tier_i >= minimum_required_tier:
      verified_gpu_hours_i = gpu_count_i × usable_hours_i
  else:
      exclude rental and publish exclusion_reason

no hidden multiplier:
  verification changes admission status, not volume weight
Uptime measurement standard: uptime_ratio_i must come from independently observable telemetry, third-party monitoring, customer-side job evidence, or cryptographic/hardware attestation. Provider self-reported uptime alone is not sufficient for production VWAP admission.
05 · True VWAP

Completed Rental VWAP is total verified spend divided by total verified GPU-hours.

Completed Rental VWAP
for each admitted completed rental i:

price_i =
  total_paid_usd_i / verified_gpu_hours_i

volume_i =
  verified_gpu_hours_i

completed_rental_vwap =
  Σ(price_i × volume_i) / Σ(volume_i)

equivalent:
  completed_rental_vwap =
  Σ(total_paid_usd_i) / Σ(verified_gpu_hours_i)
Thin-market caveat: if a small number of admitted rentals dominate the denominator, the published value may be mathematically correct but economically misleading. The index must publish concentration, rental type mix, staleness, and settlement readiness.
06 · Thin-market controls

Do not let one large deal define the market.

ControlRuleWhy it matters
Rental type stratificationCalculate spot, on-demand, bulk, reserved, and enterprise slices separately.Prevents negotiated discounts from poisoning the on-demand index.
Minimum countRequire minimum completed rentals and minimum unique providers before settlement readiness.Stops thin, decorative price prints from becoming benchmarks.
Concentration capNo provider, buyer, or buyer-provider pair can exceed a defined share of volume.Limits dominance by one actor or one strategic deal.

Carryforward ruleIf no valid rentals occur, carry prior value for max 4 hours. Flag after 60 minutes. Publish null after max age.Prevents false precision during dead windows.
07 · Manipulation model

Wash leases are more dangerous than fake offers.

Fake offers can be excluded from VWAP. Fake completed rentals are harder: an attacker can control both buyer and provider, create low-priced transactions, and attempt to move the completed-rental index.

08 · Offer Index

Offers are useful, but only as a separate supply instrument.

Verified Offer Index
filter stage:
  reject if below sanity floor
  reject if stale
  reject if telemetry fails
  reject if provider in quarantine
  reject if source cap exceeded

quality score:
  raw_quality_score_i =
    0.40 × independent_uptime_score_i
  + 0.30 × hardware_verification_score_i
  + 0.20 × provider_reputation_score_i
  + 0.10 × freshness_score_i

bounded_quality_score_i =
  clamp(raw_quality_score_i, 0.10, 1.00)

weighting stage:
  included_offer_weight_i =
    available_gpu_hours_i × bounded_quality_score_i

verified_offer_index =
  weighted_median(offer_price_i, included_offer_weight_i)
Design principle: avoid a pure multiplicative chain where one low factor collapses weight to zero. Use hard filters first, then bounded weights for included offers. The 0.10 floor preserves new-entrant diversity among offers that already passed the admission filters, while obvious manipulation is still rejected before weighting.
09 · Web2 Anchor

Web2 anchors context. It should not suppress the DePIN signal.

Use

Web2 cloud and neocloud pricing provides a stable comparison baseline and enables the DePIN Discount Spread.

Limit

Public list prices are slow-moving, may omit private discounts, and do not represent transaction-weighted market clearing prices.

DePIN Discount Spread
depin_discount_spread =
  1 - (completed_rental_vwap / web2_cloud_anchor)

example:
  1 - (2.85 / 4.20) = 32.1%

12 · Governance

Methodology changes need a public process.

Even before formal benchmark regulation applies, Silicon Oracle should act like a benchmark administrator: methodology changes are versioned, announced, logged, and never silently retrofitted into historical prices.

Governance controlPolicyWhy it matters
VersioningEvery formula, threshold, source rule, and exclusion rule belongs to a methodology version.Users know which rules produced each price.
Change noticeMaterial methodology changes require at least 14 days public notice before activation.Protocols and funds can evaluate impact before relying on the feed.
Emergency policyEmergency changes may activate immediately only to protect benchmark integrity, with immediate disclosure, rationale, and post-hoc review.Prevents silent discretionary benchmark changes.
Restatement policyHistorical values are restated only under documented criteria, with prior and new values preserved.Creates predictable behavior when bad data is discovered later.
13 · Data rights

Source rights are part of benchmark quality.

Commercial publication rule: Silicon Oracle should obtain formal data licensing agreements with source marketplaces before commercial redistribution of off-chain marketplace data. Public on-chain records may be cleaner, but every source still receives a rights status.

17 · Final position

The credible product is the benchmark stack, not the blend.

Silicon Oracle should launch with component series: DePIN Lease VWAP, Web2 Cloud Anchor, Verified Offer Index, and DePIN Discount Spread. The composite can exist as an experimental field, but the product earns trust by using binary verification admission, independently measured uptime, explicit carryforward limits, wash-lease defenses, rental-type stratification, bounded offer-quality weights, data-rights statuses, and public methodology governance.

Concise call framing: The product should not ask users to trust a blended number first. It should show the data stack that would eventually justify one.
18 · Next steps

From white paper to a daily-published benchmark.

All four component series are buildable today with rights-clean data. The DePIN Lease VWAP comes from public on-chain leases, the Web2 Cloud Anchor from published cloud and neocloud list prices, the Verified Offer Index from filtered supply-side offers, and the DePIN Discount Spread is derived from the two. The one input we cannot source yet — transaction-weighted Web2 executed prices — is marked pending and built toward in Phase 2 via licensing, not scraping.
